# Pokemon GO 自动交易（按键精灵手机助手）

这个仓库包含一个按键精灵手机助手脚本：`auto_trade.mq`。设备基准：iPhone 13 Pro / iOS 16.3，竖屏。

## 这次修了什么：为什么会“选完第一只宝可梦就卡住、不点 NEXT”

旧版脚本自己用 `GetPixelColor` 写了逐像素双重循环来扫描按钮区域：

- 每个采样点要调用 3 次 `GetPixelColor`（取 R、G、B 各一次）；
- 扫一遍 NEXT 区域约是 8000+ 个采样点、两万多次屏幕取色调用；
- 在手机助手上每次取色都是一次较慢的屏幕查询，一遍扫描要几分钟到几十分钟；
- 而且日志在整遍扫完之前不会打印。

NEXT 恰好是流程里**第一个**用颜色扫描的步骤（LOCAL TRADE、选第一只宝可梦都是普通坐标点击），所以脚本看起来就是“选完宝可梦后死在那里，不点 NEXT”。它不是逻辑卡死，是扫描慢到等于卡死。

## 现在的做法

按开发文档（help.anjian.com）改用**原生区域找色命令**，一次调用由引擎毫秒级完成整个区域扫描：

```
FindColor 左上X, 左上Y, 右下X, 右下Y, "BGR颜色-偏色", 方向, 相似度, intX, intY
// 未找到时 intX / intY = -1；颜色是 BGR 十六进制，支持 "颜色-偏色" 容差
```

在这之上做了防误点验证：

1. **长条按钮双重验证**（NEXT / CONFIRM / 橙色 CANCEL）：找到候选色点后，再验证其右侧 36~170px、下方 45~90px 仍是同色。文字（比如交易页中间的金色“+4 / xxx km”、卡片里的青色地名）没有这种横向+纵向厚度，过不了验证。验证失败的候选点会被跳过，搜索从它下方继续（`FindColor` 方向 0 只返回区域里最上面的匹配点，不续搜的话一个同色杂点就会一直挡住下方的真按钮）；完成页 X 的检测同理。
2. **位置约束**：CONFIRM/CANCEL 的色点必须出现在屏幕左侧 25% 内（按钮贴左边）；NEXT 的色点必须在左侧 35% 内。右下角自己的宝可梦卡片等不会误判。
3. **状态机顺序**：完成页的 X 只在“橙色 CANCEL 消失”（=双方都确认、动画开始）之后才开始找，所以交易页/好友页阶段绝不会去点底部区域。
4. **每步都验证**：点 NEXT 后验证 NEXT 消失才进下一步；点 CONFIRM 后验证变橙才算确认成功；点 X 后在点击点附近复查 X 消失才算关闭。任何一步限时内没达到预期状态，打日志后安全退出，不盲点。

## 默认流程

从好友资料页开始运行（能看到 `LOCAL TRADE` / `REMOTE TRADE` 的页面）：

1. 点击 `LOCAL TRADE`（坐标点击，实测可用，保留）。
2. 等宝可梦列表加载，点左上第一只宝可梦（坐标点击，保留）。
3. 扫描绿色 `NEXT`，一出现就点；点后验证页面切走，没切走限时重点。若 NEXT 长时间不出现，每 10 秒补点一次第一只宝可梦（兼容列表加载慢）。
4. 交易确认页：等对方选好宝可梦、左侧绿色 `CONFIRM` 出现就点；直到左侧变**橙色 CANCEL** 才算确认成功（没生效会自动再点）。
5. 橙色 CANCEL 在屏幕上时耐心等对方确认（只检测不乱点）；橙色连续两次检测不到 = 双方确认完毕、交易动画开始。
6. 等动画放完、收到的宝可梦详情页出现，点底部中间浅青色 `X`，复查确认它消失。
7. 等回到好友页，进入下一轮。每轮开始前还会检查上一轮的 X 是否残留（防止入口坐标误点到详情页的 POWER UP / EVOLVE）。

## 可调参数（都在 `auto_trade.mq` 顶部）

| 变量 | 默认 | 含义 |
| --- | --- | --- |
| `MaxTrades` | 20 | 最多交易轮数，0 = 无限 |
| `WaitPokemonListSec` | 12 | 点交易入口后等列表的秒数 |
| `SelectRetryIntervalSec` | 10 | NEXT 没出现时补点第一只宝可梦的间隔 |
| `WaitNextMaxSec` | 120 | 等 NEXT 出现的上限（一出现就点） |
| `NextRetryMaxSec` | 45 | NEXT 点了没切页时持续重点的上限 |
| `WaitPartnerSelectSec` | 240 | 等对方选好 + CONFIRM 生效（变橙）的上限 |
| `WaitPartnerConfirmSec` | 300 | 我方确认后等对方确认的上限 |
| `WaitCompleteMaxSec` | 120 | 双方确认后等动画结束、X 出现的上限 |
| `WaitReturnSec` | 10 | 关掉 X 后等回到好友页的秒数 |

颜色与区域也集中在脚本顶部：`GreenBtnSpec` / `OrangeSpecA、B` / `CloseSpecA、B`（BGR-偏色），以及 `Next* / Conf* / Close*` 检测区域（按屏幕百分比写，分辨率无关；iPhone 是否开“放大显示”都不影响）。橙色和浅青色各给了 BGR 与 R/B 互换两套写法同时检测，兼容引擎对颜色串按 RGB 解析的情况；绿色按钮的取值本身 R≈B 对称，天然兼容。

## 已知边界

- 如果**被交易的宝可梦本身是大面积黄绿色**（如绿毛虫系），它的模型若恰好伸进 CONFIRM 检测带（屏幕左侧 38%~59% 高度），可能被误判成绿色按钮；脚本点了它没变橙会自动重试并最终安全退出，不会乱点其他按钮。遇到这种情况把 `ConfY2` 再调小一点即可。
- 特殊交易（幸运/异色/未登录图鉴）有额外弹窗，不在本脚本处理范围内；脚本会因等不到预期状态而安全退出。
- 若某一步识别不到，优先在按键精灵 IDE 里取一次实际按钮颜色，微调对应的 `*Spec` 偏色范围，不需要改流程逻辑。
