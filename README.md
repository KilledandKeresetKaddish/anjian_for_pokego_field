# Pokemon GO 自动交易（按键精灵手机助手）

这个仓库包含一个按键精灵手机助手脚本：`auto_trade.mq`。设备基准：iPhone 13 Pro / iOS 16.3，竖屏。所有检测区域按屏幕百分比写，分辨率无关，方便适配其他尺寸的 iPhone。

**当前版本：v7.0**（版本号写在脚本文件头部注释里，启动日志第一行也会打印 `版本 v7.0`——每次测试前先核对日志里的版本号，避免误测旧版，这事出过一次）。

## 核心做法

按开发文档（help.anjian.com）用**原生区域找色命令**做状态检测，一次调用由引擎毫秒级完成整个区域扫描（这也是第一版“选完宝可梦卡死”的修复：旧版自己用 `GetPixelColor` 逐像素双重循环，一遍要扫几万次、几分钟到几十分钟，看起来就是卡死）：

```
FindColor 左上X, 左上Y, 右下X, 右下Y, "BGR颜色-偏色", 方向, 相似度, intX, intY
// 未找到时 intX / intY = -1；颜色是 BGR 十六进制（实测确认，取色器显示的也是 BGR）
```

整个流程是状态机：**等待状态出现 → 点击 → 验证状态切换**，几乎没有固定硬等；任何一步限时内等不到预期状态就打日志安全退出，绝不盲点。

### 防误点设计（多轮实测迭代）

1. **长条按钮三重验证**（NEXT / CONFIRM / 橙色 CANCEL / 详情页 POWER UP）：色点必须贴左（位置约束）+ 右侧 150~280px 仍是同色（真按钮都是 340px+ 的长条；好友页“64 LEVEL”两行青色文字宽约 95px 够不着）+ 下方 45~90px 仍是同色（单行文字没有这个纵向厚度）。验证失败的候选点会被跳过、从它下方继续搜索（`FindColor` 只返回区域里最上面的匹配点，不续搜的话一个杂点就会一直挡住真按钮）。
2. **颜色只按 BGR 一种口径**。不能为“兼容 RGB 解析”同时检测 R/B 互换色：橙色互换后就是蓝色，交易页大面积蓝背景会整页误中（实测翻车，已删除）。
3. **“详情页开着”用双探针页面标记判断**：左侧 60%~84% 高度的贴左绿色长按钮（POWER UP / EVOLVE / CHANGE FORM），或右侧 74%~86%、高度 48%~56% 的**纵向白色长列**兜底——白点正下方 25~160px 分四段窗口必须全白（按钮可能因星尘/糖果不足变灰、满级消失，白卡不依赖按钮状态）。这个标记的硬性要求是**在好友页上必须为 False**，否则脚本会点掉好友页自己的 X：v6 的白卡探针只验证一小段就误报过（好友页 48%~56% 是人物展示场景，云朵/白色伙伴宝可梦/白色衣物也能撑过一小段白色窗口），v7 改为纵向长列后场景元素过不全（其下缘是天空蓝/地面色）。交易完成收尾全部以标记为准：标记在才找 X、标记连续消失才算关闭成功。每轮开始前会打印两个探针的诊断（`标记诊断[轮前]`），**只诊断、不点击**。
4. **X 圆钮判定**：候选点“正下方窄列”25~85px 必须仍是同色——X 是实心青色圆钮有纵向厚度，GYMS & RAIDS 页签的细下划线正下方是白底，一票否决；左右 130px 以外不允许还有同色，排除长按钮。找不到 X 超过 8 秒时启动比例兜底：x 固定实测的 50.5%，高度按 93.5%/90.5%/96%/87.5% 四档轮换（详情页 X 高度从截图只能标定到 ±3%，轮换保证覆盖），点没点中由“详情页标记是否消失”说了算。v7 加保险丝：识别到的 X 最多点 3 次、比例兜底最多 4 档，全部用完标记还在就打印双探针诊断并安全退出，绝不无限补点。

## 默认流程

从好友资料页开始运行（能看到 `LOCAL TRADE` / `REMOTE TRADE` 的页面）：

1. 轮前诊断：打印详情页标记两个探针的状态（`标记诊断[轮前]`），**只诊断不点击**——v7 起开局绝不碰任何 X（v5/v6 都发生过标记在好友页误报、把好友页自己的 X 点掉的事故）。若标记复测仍成立会告警继续；详情页真没关时本轮会因等不到 NEXT 安全超时退出。
2. 点 `LOCAL TRADE`，短缓冲 4 秒后点左上第一只宝可梦；列表加载慢没关系——NEXT 没出现会每 6 秒自动重选一次。
3. 绿色 `NEXT` 一出现就点；点后验证页面切走，没切走限时重点。
4. 等对方选好、左侧绿色 `CONFIRM` 出现就点；直到左侧变**橙色 CANCEL** 才算确认成功（对方先确认导致橙色一闪而过的情况有兜底：点过之后绿/橙都连续消失也算交易开始）。
5. 橙色 CANCEL 在屏幕上时耐心等对方确认（只检测不乱点）；橙色连续两次检测不到 = 双方确认、交易动画开始。
6. 等“详情页标记”出现（动画放完），在底部带找 X 圆钮点掉；验证**标记消失**才算关闭成功。
7. 短缓冲 5 秒回好友页，进入下一轮（下一轮开头打印标记诊断，只看不点）。

## 可调参数（都在 `auto_trade.mq` 顶部）

| 变量 | 默认 | 含义 |
| --- | --- | --- |
| `MaxTrades` | 20 | 最多交易轮数，0 = 无限 |
| `ListSettleSec` | 4 | 点交易入口后的短缓冲（列表慢会自动重选，不用调大） |
| `SelectRetryIntervalSec` | 6 | NEXT 没出现时补点第一只宝可梦的间隔 |
| `WaitNextMaxSec` | 120 | 等 NEXT 出现的上限（一出现就点） |
| `NextRetryMaxSec` | 45 | NEXT 点了没切页时持续重点的上限 |
| `WaitPartnerSelectSec` | 240 | 等对方选好 + CONFIRM 生效（变橙）的上限 |
| `WaitPartnerConfirmSec` | 300 | 我方确认后等对方确认的上限 |
| `WaitCompleteMaxSec` | 150 | 双方确认后：动画 + 详情页出现 + X 点掉 + 标记消失的总上限 |
| `ReturnSettleSec` | 5 | 详情页确认关闭后回好友页的短缓冲 |

颜色集中在脚本顶部：`GreenBtnSpec` / `OrangeSpec` / `CloseSpec`（BGR-偏色）；区域在 `Next* / Conf* / Close* / Det*`。

## 实测修复记录

**第一轮：选完第一只宝可梦卡死** —— 旧版逐像素 `GetPixelColor` 扫描慢到等于卡死；改原生 `FindColor`。

**第二轮：两个误判** —— ① R/B 互换的“兼容”橙色就是蓝色，交易页蓝背景被当成“已确认”：删互换色、只用 BGR。② Furfrou 的 CHANGE FORM 长按钮落在 X 检测带里被点中：收窄检测带、加排长条验证。

**第三轮：X 点不到 + 好友页同款 X 导致卡死** ——
① 日志显示一直点 (385,1874)，而实测 X 中心在 (485,1949)：抢走候选点的是 GYMS & RAIDS 页签的青色下划线（恰好压在检测带顶上），它通过了旧的宽框验证（框边擦到了旁边 X 的边缘像素）。修复：改“正下方窄列”验证，细线正下方是白底，直接否决；X 颜色按实测取色 RGB(96,205,176)（BGR `B0CD60`）重新放宽（旧 G 上限 196 连 X 笔画都排除了）。
② “点完 X 后在原位置复查 X 是否消失”这个验证逻辑本身是错的：好友页底部也有一颗同款 X（实测同位置），关掉详情页回到好友页永远“X 还在”，导致第二轮开局误判残留、安全退出。修复：验证一律改用“详情页标记”（见上），好友页上标记为 False，不会再碰任何 X。
③ 硬等待全面改为检测驱动：列表 12 秒硬等 → 4 秒缓冲 + NEXT 驱动自动重选；回好友页 10 秒硬等 → 标记消失即走 + 5 秒缓冲；每轮固定开销从 ~23.5 秒降到 ~10 秒。

**第四轮：覆盖详情页 X 高度的标定误差** —— 详情页的 X 与好友页实测点 (485,1949)=93.7% 可能不完全同高（截图裁切只能标定到 ±3%）。X 检测带放宽到 84%~98%；兜底点击改为四档高度轮换；X 识别失败时日志会打印带内青色原始点坐标，便于直接校准。

**第五轮：开局误点好友页的 X** —— 好友页“64 LEVEL”两行青色文字（贴左、“64”宽约 95px、下方还有第二行）骗过了旧的 60px 横向验证，让“详情页标记”在好友页误报，开局 PreEntryGuard 直接点掉了好友页的 X。修复：横向验证加严到 150~280px（真按钮都 ≥340px，文字够不着）；PreEntryGuard 改为 800ms 后复测仍成立才动手。

**第六轮：详情页标记不依赖按钮状态（codex review 指出）** —— POWER UP 因星尘/糖果不足变灰、或宝可梦满级时，左侧可能没有任何可识别的绿色按钮，旧标记会一直为 False，X 永远不会被点，最终带着没关的详情页超时退出。修复：新增右侧纯白大卡片探针（74%~86%、48%~56%，避开右缘悬浮窗）作为标记的并联条件；白卡不依赖按钮状态。（注：当时认为“好友页同位置是天空蓝，不会误报”，第七轮实测推翻了这个假设。）

**第七轮（v7.0）：开局又把好友页误判成详情页 + 防线重构** —— 第六轮的白卡探针只验证“白点下方 40~90px 仍是白色”，而好友页 48%~56% 这一带是人物展示场景：云朵/白色伙伴宝可梦/白色衣物同样能撑过一小段白色窗口。结果 PreEntryGuard 开局误判“详情页未关闭”、连续补点，把好友页自己的 X 点掉（实测日志：补点 @495,1939、@502,1939，第三次走比例兜底 @484,1819），整轮报废。修复分三层，并且每层都假设“探针可能再次说谎”：

1. **白卡探针加严为“纵向白色长列”**：白点正下方 25~160px 分四段窗口必须全部仍是白色。真详情页的白卡占满整个探测带（48%~56% 全是纯白、白点几乎都在带顶），160px 长列仍落在带内，不会漏判；好友页场景元素的下缘是天空蓝/地面色，四段过不全。验证窗右缘收在探测带内，防止擦到 x≥87% 的白色 LiveFeed 悬浮窗。
2. **轮前复查降级为“只诊断、绝不点击”**：每轮开始打印两个探针各自的命中情况和坐标（`标记诊断[轮前]：绿长钮=…，白卡列=…`），标记成立时 800ms 后复测、告警，但不再做任何补点动作。它原本想兜底的“上一轮详情页残留”在收尾步骤已有“标记连续两次消失才算成功、失败即停脚本”的验证，几乎不可能漏到下一轮；而误报补点的代价（点掉好友页的 X）是毁灭性的。真有残留时，本轮会因等不到 NEXT 安全超时退出。
3. **收尾补点加保险丝**：识别到的 X 最多点 3 次、比例兜底最多轮 4 档；全部用完标记还在，大概率是标记在当前页面误报（比如点掉真 X 回到好友页后探针又说谎——旧版会接着把好友页、好友列表一路点穿），此时打印双探针诊断并安全退出，把误报现场留在屏幕上。

同时给脚本加了版本号：文件头注释 + 启动日志（`Auto trade start. 版本 v7.0 …`），测试前先核对，防止再误测旧版。若开局仍出现误报，把日志里的 `标记诊断[轮前]` 两行（含坐标）和当时的屏幕截图发回即可精确定位是哪个探针、在哪个坐标说谎。

## 已知边界

- 如果**被交易的宝可梦本身是大面积黄绿色**（如绿毛虫系），模型恰好伸进 CONFIRM 检测带（左侧 38%~59% 高度）可能被误判成按钮；点了没变橙会自动重试并最终安全退出。把 `ConfY2` 调小即可。
- 特殊交易（幸运/异色/未登录图鉴）有额外弹窗，不在处理范围内；脚本会因等不到预期状态安全退出。
- 某一步识别不到时，优先用 IDE 取色器取实际颜色（注意显示的是 BGR），微调对应 `*Spec`；位置不对就调对应区域百分比。日志里每一步都有“等什么、等了多久”，能直接定位卡点。
