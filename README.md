# Pokemon GO 自动交易（按键精灵手机助手）

这个仓库包含一个按键精灵手机助手脚本：`auto_trade.mq`。设备基准：iPhone 13 Pro / iOS 16.3，竖屏。所有检测区域按屏幕百分比写，分辨率无关，方便适配其他尺寸的 iPhone。

## 核心做法

按开发文档（help.anjian.com）用**原生区域找色命令**做状态检测，一次调用由引擎毫秒级完成整个区域扫描（这也是第一版“选完宝可梦卡死”的修复：旧版自己用 `GetPixelColor` 逐像素双重循环，一遍要扫几万次、几分钟到几十分钟，看起来就是卡死）：

```
FindColor 左上X, 左上Y, 右下X, 右下Y, "BGR颜色-偏色", 方向, 相似度, intX, intY
// 未找到时 intX / intY = -1；颜色是 BGR 十六进制（实测确认，取色器显示的也是 BGR）
```

整个流程是状态机：**等待状态出现 → 点击 → 验证状态切换**，几乎没有固定硬等；任何一步限时内等不到预期状态就打日志安全退出，绝不盲点。

### 防误点设计（多轮实测迭代）

1. **长条按钮三重验证**（NEXT / CONFIRM / 橙色 CANCEL / 详情页 POWER UP）：色点必须贴左（位置约束）+ 右侧 60~170px 仍是同色（“64”等级数字、金色“+4”这类 ~55px 宽的文本过不了）+ 下方 45~90px 仍是同色（文字行没有这个纵向厚度）。验证失败的候选点会被跳过、从它下方继续搜索（`FindColor` 只返回区域里最上面的匹配点，不续搜的话一个杂点就会一直挡住真按钮）。
2. **颜色只按 BGR 一种口径**。不能为“兼容 RGB 解析”同时检测 R/B 互换色：橙色互换后就是蓝色，交易页大面积蓝背景会整页误中（实测翻车，已删除）。
3. **“详情页开着”用页面标记判断**：左侧 60%~84% 高度的贴左绿色长按钮（POWER UP / EVOLVE / CHANGE FORM，每个宝可梦详情页必有 POWER UP）。交易完成收尾全部以它为准：标记在才找 X、标记连续消失才算关闭成功、每轮开始前复查标记防残留。
4. **X 圆钮判定**：候选点“正下方窄列”25~85px 必须仍是同色——X 是实心青色圆钮有纵向厚度，GYMS & RAIDS 页签的细下划线正下方是白底，一票否决；左右 130px 以外不允许还有同色，排除长按钮。找不到 X 超过 12 秒时按实测比例坐标（50.5%, 93.7%）兜底点一次。

## 默认流程

从好友资料页开始运行（能看到 `LOCAL TRADE` / `REMOTE TRADE` 的页面）：

1. 轮前复查：详情页标记还在就先补点 X 关闭（防上一轮残留误点 POWER UP）。
2. 点 `LOCAL TRADE`，短缓冲 4 秒后点左上第一只宝可梦；列表加载慢没关系——NEXT 没出现会每 6 秒自动重选一次。
3. 绿色 `NEXT` 一出现就点；点后验证页面切走，没切走限时重点。
4. 等对方选好、左侧绿色 `CONFIRM` 出现就点；直到左侧变**橙色 CANCEL** 才算确认成功（对方先确认导致橙色一闪而过的情况有兜底：点过之后绿/橙都连续消失也算交易开始）。
5. 橙色 CANCEL 在屏幕上时耐心等对方确认（只检测不乱点）；橙色连续两次检测不到 = 双方确认、交易动画开始。
6. 等“详情页标记”出现（动画放完），在底部带找 X 圆钮点掉；验证**标记消失**才算关闭成功。
7. 短缓冲 5 秒回好友页，进入下一轮（下一轮开头还有标记复查兜底）。

## 可调参数（都在 `auto_trade.mq` 顶部）

| 变量 | 默认 | 含义 |
| --- | --- | --- |
| `MaxTrades` | 20 | 最多交易轮数，0 = 无限 |
| `ListSettleSec` | 4 | 点交易入口后的短缓冲（列表慢会自动重选，不用调大） |
| `SelectRetryIntervalSec` | 6 | NEXT 没出现时补点第一只宝可梦的间隔 |
| `WaitNextMaxSec` | 120 | 等 NEXT 出现的上限（一出现就点） |
| `NextRetryMaxSec` | 45 | NEXT 点了没切页时持续重点的上限 |
| `WaitPartnerSelectSec` | 240 | 等对方选好 + CONFIRM 生效（变橙）的上限 |
| `WaitPartnerConfirmSec` | 300 | 我方确认后等对方确认的上限 |
| `WaitCompleteMaxSec` | 150 | 双方确认后：动画 + 详情页出现 + X 点掉 + 标记消失的总上限 |
| `ReturnSettleSec` | 5 | 详情页确认关闭后回好友页的短缓冲 |

颜色集中在脚本顶部：`GreenBtnSpec` / `OrangeSpec` / `CloseSpec`（BGR-偏色）；区域在 `Next* / Conf* / Close* / Det*`。

## 实测修复记录

**第一轮：选完第一只宝可梦卡死** —— 旧版逐像素 `GetPixelColor` 扫描慢到等于卡死；改原生 `FindColor`。

**第二轮：两个误判** —— ① R/B 互换的“兼容”橙色就是蓝色，交易页蓝背景被当成“已确认”：删互换色、只用 BGR。② Furfrou 的 CHANGE FORM 长按钮落在 X 检测带里被点中：收窄检测带、加排长条验证。

**第三轮：X 点不到 + 好友页同款 X 导致卡死** ——
① 日志显示一直点 (385,1874)，而实测 X 中心在 (485,1949)：抢走候选点的是 GYMS & RAIDS 页签的青色下划线（恰好压在检测带顶上），它通过了旧的宽框验证（框边擦到了旁边 X 的边缘像素）。修复：改“正下方窄列”验证，细线正下方是白底，直接否决；X 颜色按实测取色 RGB(96,205,176)（BGR `B0CD60`）重新放宽（旧 G 上限 196 连 X 笔画都排除了）。
② “点完 X 后在原位置复查 X 是否消失”这个验证逻辑本身是错的：好友页底部也有一颗同款 X（实测同位置），关掉详情页回到好友页永远“X 还在”，导致第二轮开局误判残留、安全退出。修复：验证一律改用“详情页标记”（见上），好友页上标记为 False，不会再碰任何 X。
③ 硬等待全面改为检测驱动：列表 12 秒硬等 → 4 秒缓冲 + NEXT 驱动自动重选；回好友页 10 秒硬等 → 标记消失即走 + 5 秒缓冲；每轮固定开销从 ~23.5 秒降到 ~10 秒。

## 已知边界

- 如果**被交易的宝可梦本身是大面积黄绿色**（如绿毛虫系），模型恰好伸进 CONFIRM 检测带（左侧 38%~59% 高度）可能被误判成按钮；点了没变橙会自动重试并最终安全退出。把 `ConfY2` 调小即可。
- 特殊交易（幸运/异色/未登录图鉴）有额外弹窗，不在处理范围内；脚本会因等不到预期状态安全退出。
- 某一步识别不到时，优先用 IDE 取色器取实际颜色（注意显示的是 BGR），微调对应 `*Spec`；位置不对就调对应区域百分比。日志里每一步都有“等什么、等了多久”，能直接定位卡点。
