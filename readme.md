| 设备 | 角色 |

- 上新机型/新界面：先各跑一轮验证，再批量使用。

## 下次不要再犯的错（每条都有真实事故背书）

### 写代码层面
1. **不要用 `GetPixelColor` 逐像素扫区域**——一遍几万次取色等于卡死（v1~v4 的根因）。区域找色只用原生 `FindColor`。
2. **不要做 RGB"兼容"或 R/B 互换检测**——橙色互换后就是蓝色，交易页蓝背景整页误中。只按 BGR 一种口径。
3. **候选点验证失败必须从它下方续搜**——否则一个杂点永远挡住真按钮。
4. **关键按钮（NEXT/CONFIRM/X）不许固定坐标盲点**，必须"等状态 → 点 → 验证切换"；固定坐标只许用于点错也无害的位置。"验证切换"也不能只看"带内同色还没消失"——点击后的失败重点必须同位复核（同一按钮还在原位 + 原位白字仍在），否则换页后站进检测带的绿色宝可梦模型会顶替 NEXT 触发盲目重点（v7.7 实测翻车）。
5. **单点/单段小窗的颜色验证必被骗**（青色文字、功能图标、页签下划线、抗锯齿灰、白底好友页、绿色宝可梦模型整块实心绿全都骗过）——必须组合结构验证：实心连续多窗 + 纵向厚度 + 总长 + 排长条 + 位置约束 + 贴左/白字（交易页按钮从屏幕左缘伸出且印纯白文字，模型两样都没有）。

### 改检测逻辑层面
6. **不要凭猜改色盒**——v7.4 (PR#10) 没有任何实测色值就换 spec，合并 12 分钟被整体回滚。修复必须有用户实测的色值/坐标支撑。
7. **不要为修 A 处改共用色盒殃及 B 处**——正确做法是新增专用 spec（v7.6 加 `XSolidSpec` 而一字不动 `CloseSpec`，因为后者还在好友页锚点并集里用）。
8. **色盒要排中性灰**：文字抗锯齿灰（R≈G≈B，132~155）是惯犯。关键判定用「G≥178 且 R≤142」这类数学上排除 R=G=B 的写法。
9. **不要破坏安全不变量**，任何改动前论证它们仍成立：
   - 详情页标记「在好友页恒为 False」；
   - 好友页锚点「在详情页恒不成立」（绿长钮硬安全闸保证）；
   - 等对方确认期间绝不重复点 CONFIRM 位置（会点到 CANCEL，交易作废）；
   - 弹窗上绝不点主按钮（那是 YES = 真进化/强化，不可逆），只点下方 NO；
   - 轮前全程绝不补点 X（点掉好友页自己的 X 翻过三次车）。
10. **风险分级响应**：检出绿色长按钮 = 可能误进化/耗星尘 → 硬停交人工；无副作用的疑似情况 → 告警放行交超时兜底。一刀切两头（只告警继续 / 确认即硬停）都翻过车。
11. **一切重试/补点都加保险丝**（识别 X ≤3 次、兜底 ≤4 档、弹窗 ≤4 次这类），用完打诊断安全退出，绝不无限重试。
12. **UI 随时会改版**（图标改彩色插画打穿过锚点、好友页出过白底样式）——检测逻辑必须配可校准的分解诊断日志，失败时一行日志能看出哪个窗、哪个色谱没命中。

### 流程/文档层面
13. **每次改脚本必须 bump 版本号**（脚本头注释 + 启动日志第一行），并提醒用户测试前核对——误测旧版出过事故。
14. **PR 描述和文档写准史实**：分支带了什么 commit 就写什么（PR#9 实际带 v7.2+v7.3 但描述只写 v7.2；"v7.4 未合并"其实是合并后被回滚）。
15. **先要证据再动手**：出问题先要日志（每步都打"等什么、等了多久"）和截图，对照标定后再修；排障路径 = 日志定位卡点 → 取色器实测（BGR）→ 调对应 `*Spec` 或区域百分比。
16. **方向错了整体 revert 再重做**（#11 回滚 #10、#12 重做），不要在错误方案上打补丁。
