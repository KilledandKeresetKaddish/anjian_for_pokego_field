# Pokemon GO 自动交易（按键精灵手机助手）

这个仓库包含一个按键精灵手机助手脚本：`auto_trade.mq`。设备基准：iPhone 13 Pro / iOS 16.3，竖屏。所有检测区域按屏幕百分比写，分辨率无关，方便适配其他尺寸的 iPhone。

**当前版本：v7.5**（版本号写在脚本文件头部注释里，启动日志第一行也会打印 `版本 v7.5`——每次测试前先核对日志里的版本号，避免误测旧版，这事出过一次。v7.4 是 PR#10 的未合并尝试，分析出三处问题后被 v7.5 取代，见第十轮记录）。

## 核心做法

按开发文档（help.anjian.com）用**原生区域找色命令**做状态检测，一次调用由引擎毫秒级完成整个区域扫描（这也是第一版“选完宝可梦卡死”的修复：旧版自己用 `GetPixelColor` 逐像素双重循环，一遍要扫几万次、几分钟到几十分钟，看起来就是卡死）：

```
FindColor 左上X, 左上Y, 右下X, 右下Y, "BGR颜色-偏色", 方向, 相似度, intX, intY
// 未找到时 intX / intY = -1；颜色是 BGR 十六进制（实测确认，取色器显示的也是 BGR）
```

整个流程是状态机：**等待状态出现 → 点击 → 验证状态切换**，几乎没有固定硬等；任何一步限时内等不到预期状态就打日志安全退出，绝不盲点。

### 防误点设计（多轮实测迭代）

1. **长条按钮四重验证**（NEXT / CONFIRM / 橙色 CANCEL / 详情页 POWER UP）：色点必须贴左（位置约束）+ **实心连续五连窗**（候选点右侧 +20~+215 每 40px 一个 35px 窗口，每个窗口都必须仍有同色——真按钮是 340px+ 的实心长条全过；好友页四枚青色功能图标单枚 ~75px、间距 ~150px，空隙必让某个整窗落空，v7.2 加，堵住“+150~280 恰好撞上相邻图标”的漏洞）+ 右侧 150~280px 仍是同色（总长证明；“64 LEVEL”两行青色文字宽约 95px 够不着）+ 下方 45~90px 仍是同色（单行文字没有这个纵向厚度）。验证失败的候选点会被跳过、从它下方继续搜索（`FindColor` 只返回区域里最上面的匹配点，不续搜的话一个杂点就会一直挡住真按钮）。
2. **颜色只按 BGR 一种口径**。不能为“兼容 RGB 解析”同时检测 R/B 互换色：橙色互换后就是蓝色，交易页大面积蓝背景会整页误中（实测翻车，已删除）。
3. **“详情页开着”用双探针页面标记 + 好友页锚点一票否决判断**：左侧 60%~84% 高度的贴左绿色长按钮（POWER UP / EVOLVE / CHANGE FORM），或右侧 74%~86%、高度 48%~56% 的**纵向白色长列**兜底——白点正下方 25~160px 分四段窗口必须全白（按钮可能因星尘/糖果不足变灰、满级消失，白卡不依赖按钮状态）。这个标记的硬性要求是**在好友页上必须为 False**，否则脚本会点掉好友页自己的 X：v6 的白卡探针只验证一小段就误报过（好友页人物场景里的云朵/白色伙伴宝可梦/白色衣物能撑过一小段白色窗口），v7 改为纵向长列；v7.2 实测又发现好友页存在**白底样式**（人物背景整片纯白 + 队徽水印），白色长列在这种样式上天然成立、白色本身已无法区分两页——因此新增**好友页锚点**：LOCAL TRADE（35%~43%）与 BATTLE（81%~86%）两枚图标窗 + 图标间净空（46%~54%）同时成立即确认当前是好友页，标记强制 False。图标窗找色 v7.5 起用**三色谱并集**（`GreenBtnSpec` 黄绿~青绿 / `CloseSpec` PoGo 青色 / `FriendBlueSpec` 饱和蓝，任一命中即算图标在场）：当前界面这排图标已是彩色插画（LOCAL TRADE 蓝手机、BATTLE 绿场景），单一色盒必然顾此失彼。图标行高度窗 73.5%~78.8%，下沿必须停在 78.8%——79.2% 起是图标文字标签，圈进来会让“REMOTE TRADE”标签左端伸进间隙窗、抗锯齿中性灰误触发净空否决。锚点另有**绿长钮硬安全闸**（v7.5）：检测带内验证出实心绿色长按钮 = 一定不是好友页，锚点直接不成立——所以“POWER UP / EVOLVE 可点的详情页”在结构上不可能被误认成好友页，色谱放宽不牺牲安全性；白卡-only 的详情页仍靠 BATTLE 窗空白区保证锚点不成立。交易完成收尾全部以标记为准：标记在才找 X、标记连续消失才算关闭成功。v7.3 起正常轮次开头不再做怀疑式预检，而是用锚点**正向确认好友页**后一行日志直接放行；锚点超时才打印**锚点分解**（逐窗×逐色谱命中情况）和两个探针 + 锚点的诊断（`标记诊断[轮前]`）做 0.8s/2s 三连确认，v7.5 起按结果分流：确认且含绿长钮 -> 安全退出（不可逆风险硬停）；只有白卡列 -> 告警后放行（无绿钮即无耗星尘/误进化风险，页面不对由 NEXT 超时兜底），**全程绝不补点 X**。
4. **X 圆钮判定**：候选点“正下方窄列”25~85px 必须仍是同色——X 是实心青色圆钮有纵向厚度，GYMS & RAIDS 页签的细下划线正下方是白底，一票否决；左右 130px 以外不允许还有同色，排除长按钮。找不到 X 超过 8 秒时启动比例兜底：x 固定实测的 50.5%，高度按 93.5%/90.5%/96%/87.5% 四档轮换（详情页 X 高度从截图只能标定到 ±3%，轮换保证覆盖），点没点中由“详情页标记是否消失”说了算。v7 加保险丝：识别到的 X 最多点 3 次、比例兜底最多 4 档，全部用完标记还在就打印双探针诊断并安全退出，绝不无限补点。

## 默认流程

从好友资料页开始运行（能看到 `LOCAL TRADE` / `REMOTE TRADE` 的页面）：

1. 轮前闸门（v7.3 判定反转，v7.5 三连确认分流）：默认相信"此刻就该在好友页"（首轮是使用要求，后续轮收尾已验证过标记消失），用好友页锚点（LOCAL TRADE + BATTLE 双图标定位，三色谱并集找色 + 绿长钮硬安全闸）做**正向确认**——锚点一出现立即放行、无固定等待，放行安全：锚点成立本身就证明入口坐标点的就是 LOCAL TRADE 图标本尊。锚点超时（默认 10 秒）才转入怀疑分支：先打一行**锚点分解**（逐窗×逐色谱，哪条腿断了直接可校准），再打两个探针 + 锚点的诊断（`标记诊断[轮前]`）、间隔 0.8s/2s 三连确认，按结果分流——确认且含**绿长钮**时直接安全退出并提示手动收尾（真残留时继续跑会让入口坐标 (386,1555) 盲点到 POWER UP / EVOLVE，可能消耗星尘甚至误进化）；确认但**只有白卡列**时告警后放行（无绿钮即无不可逆风险：真是灰按钮/满级详情页，入口坐标点上去也只是无副作用的空点，随后由 NEXT 超时安全退出；是好友页锚点漏认则本轮照常完成）；没被三连确认（如 BATTLE 变灰让锚点失效）同样告警后照常放行。**全程绝不补点 X**（v5/v6 都发生过标记在好友页误报、把好友页自己的 X 点掉的事故）。
2. 点 `LOCAL TRADE`，等"好友页锚点消失 = 已切去列表"（检测驱动，通常 1 秒内；超时自动补点一次入口）后点左上第一只宝可梦；列表加载慢没关系——NEXT 没出现会每 6 秒自动重选一次。
3. 绿色 `NEXT` 一出现就点；点后验证页面切走，没切走限时重点。
4. 等对方选好、左侧绿色 `CONFIRM` 出现就点；直到左侧变**橙色 CANCEL** 才算确认成功（对方先确认导致橙色一闪而过的情况有兜底：点过之后绿/橙都连续消失也算交易开始）。
5. 橙色 CANCEL 在屏幕上时耐心等对方确认（只检测不乱点）；橙色连续两次检测不到 = 双方确认、交易动画开始。
6. 等“详情页标记”出现（动画放完），在底部带找 X 圆钮点掉；验证**标记消失**才算关闭成功。
7. 直接进入下一轮：下一轮开头的"锚点就绪"等待就是检测驱动的回页缓冲，页面一回来立即开始（v7.3 起没有固定 5 秒等待）。

## 可调参数（都在 `auto_trade.mq` 顶部）

| 变量 | 默认 | 含义 |
| --- | --- | --- |
| `MaxTrades` | 20 | 最多交易轮数，0 = 无限 |
| `FriendReadyMaxSec` | 10 | 轮前等好友页锚点出现的上限（检测驱动，一出现立即开始；好友不在附近 BATTLE 变灰时每轮会等满，可调小） |
| `LeaveFriendMaxSec` | 8 | 点 LOCAL TRADE 后等锚点消失（已切去列表）的上限，超时自动补点一次入口 |
| `SelectRetryIntervalSec` | 6 | NEXT 没出现时补点第一只宝可梦的间隔 |
| `WaitNextMaxSec` | 120 | 等 NEXT 出现的上限（一出现就点） |
| `NextRetryMaxSec` | 45 | NEXT 点了没切页时持续重点的上限 |
| `WaitPartnerSelectSec` | 240 | 等对方选好 + CONFIRM 生效（变橙）的上限 |
| `WaitPartnerConfirmSec` | 300 | 我方确认后等对方确认的上限 |
| `WaitCompleteMaxSec` | 150 | 双方确认后：动画 + 详情页出现 + X 点掉 + 标记消失的总上限 |

颜色集中在脚本顶部：`GreenBtnSpec` / `OrangeSpec` / `CloseSpec` / `FriendBlueSpec`（BGR-偏色）；区域在 `Next* / Conf* / Close* / Det* / Fri*`。

## 实测修复记录

**第一轮：选完第一只宝可梦卡死** —— 旧版逐像素 `GetPixelColor` 扫描慢到等于卡死；改原生 `FindColor`。

**第二轮：两个误判** —— ① R/B 互换的“兼容”橙色就是蓝色，交易页蓝背景被当成“已确认”：删互换色、只用 BGR。② Furfrou 的 CHANGE FORM 长按钮落在 X 检测带里被点中：收窄检测带、加排长条验证。

**第三轮：X 点不到 + 好友页同款 X 导致卡死** ——
① 日志显示一直点 (385,1874)，而实测 X 中心在 (485,1949)：抢走候选点的是 GYMS & RAIDS 页签的青色下划线（恰好压在检测带顶上），它通过了旧的宽框验证（框边擦到了旁边 X 的边缘像素）。修复：改“正下方窄列”验证，细线正下方是白底，直接否决；X 颜色按实测取色 RGB(96,205,176)（BGR `B0CD60`）重新放宽（旧 G 上限 196 连 X 笔画都排除了）。
② “点完 X 后在原位置复查 X 是否消失”这个验证逻辑本身是错的：好友页底部也有一颗同款 X（实测同位置），关掉详情页回到好友页永远“X 还在”，导致第二轮开局误判残留、安全退出。修复：验证一律改用“详情页标记”（见上），好友页上标记为 False，不会再碰任何 X。
③ 硬等待全面改为检测驱动：列表 12 秒硬等 → 4 秒缓冲 + NEXT 驱动自动重选；回好友页 10 秒硬等 → 标记消失即走 + 5 秒缓冲；每轮固定开销从 ~23.5 秒降到 ~10 秒。

**第四轮：覆盖详情页 X 高度的标定误差** —— 详情页的 X 与好友页实测点 (485,1949)=93.7% 可能不完全同高（截图裁切只能标定到 ±3%）。X 检测带放宽到 84%~98%；兜底点击改为四档高度轮换；X 识别失败时日志会打印带内青色原始点坐标，便于直接校准。

**第五轮：开局误点好友页的 X** —— 好友页“64 LEVEL”两行青色文字（贴左、“64”宽约 95px、下方还有第二行）骗过了旧的 60px 横向验证，让“详情页标记”在好友页误报，开局 PreEntryGuard 直接点掉了好友页的 X。修复：横向验证加严到 150~280px（真按钮都 ≥340px，文字够不着）；PreEntryGuard 改为 800ms 后复测仍成立才动手。

**第六轮：详情页标记不依赖按钮状态（codex review 指出）** —— POWER UP 因星尘/糖果不足变灰、或宝可梦满级时，左侧可能没有任何可识别的绿色按钮，旧标记会一直为 False，X 永远不会被点，最终带着没关的详情页超时退出。修复：新增右侧纯白大卡片探针（74%~86%、48%~56%，避开右缘悬浮窗）作为标记的并联条件；白卡不依赖按钮状态。（注：当时认为“好友页同位置是天空蓝，不会误报”，第七轮实测推翻了这个假设。）

**第七轮（v7.0 / v7.1）：开局又把好友页误判成详情页 + 防线重构** —— 第六轮的白卡探针只验证“白点下方 40~90px 仍是白色”，而好友页 48%~56% 这一带是人物展示场景：云朵/白色伙伴宝可梦/白色衣物同样能撑过一小段白色窗口。结果 PreEntryGuard 开局误判“详情页未关闭”、连续补点，把好友页自己的 X 点掉（实测日志：补点 @495,1939、@502,1939，第三次走比例兜底 @484,1819），整轮报废。修复分三层，并且每层都假设“探针可能再次说谎”：

1. **白卡探针加严为“纵向白色长列”**：白点正下方 25~160px 分四段窗口必须全部仍是白色。真详情页的白卡占满整个探测带（48%~56% 全是纯白、白点几乎都在带顶），160px 长列仍落在带内，不会漏判；好友页场景元素的下缘是天空蓝/地面色，四段过不全。验证窗右缘收在探测带内，防止擦到 x≥87% 的白色 LiveFeed 悬浮窗。
2. **轮前复查绝不补点 X，三连确认残留即安全退出**：每轮开始打印两个探针各自的命中情况和坐标（`标记诊断[轮前]：绿长钮=…，白卡列=…`）；标记间隔 0.8s/2s 连续三次成立时停止脚本并提示手动收尾。两类事故都从机制上排除：误报补点会点掉好友页的 X（v5/v6 实测翻车）；带着真残留继续跑则会让入口坐标 (386,1555) 盲点到详情页的 POWER UP / EVOLVE，可能消耗星尘甚至误进化（v7.0 曾改成只告警就继续，codex review 指出了这个漏洞，v7.1 改为确认即退出）。
3. **收尾补点加保险丝**：识别到的 X 最多点 3 次、比例兜底最多轮 4 档；全部用完标记还在，大概率是标记在当前页面误报（比如点掉真 X 回到好友页后探针又说谎——旧版会接着把好友页、好友列表一路点穿），此时打印双探针诊断并安全退出，把误报现场留在屏幕上。

同时给脚本加了版本号：文件头注释 + 启动日志（`Auto trade start. 版本 v7.1 …`），测试前先核对，防止再误测旧版。若开局出现误报（好友页上被三连确认而退出），把日志里的三行 `标记诊断` （含坐标）和当时的屏幕截图发回即可精确定位是哪个探针、在哪个坐标说谎。

**第八轮（v7.2）：白底样式好友页让双探针一起误报、开局安全退出** —— v7.1 实测：好友页上 `标记诊断[轮前]：绿长钮=有(174,1536)，白卡列=有(718,997)` 三连确认、脚本按设计安全退出（没乱点，但一轮也没法跑）。对照截图标定出两个全新误报源：

1. **绿长钮误报源是好友页中部的四枚青色功能图标行**（SEND GIFT / LOCAL TRADE / REMOTE TRADE / BATTLE，高度 ~74%~79%）：命中点 (174,1536) 是 SEND GIFT 图标顶，图标间距 ~150px，旧验证“右侧 +150~280 仍有同色”恰好撞上相邻的 LOCAL TRADE 图标、“下方 45~90px 仍有同色”撞上图标自身下半——四重验证全过。修复：长条按钮加**实心连续五连窗**（候选点右侧 +20~+215 每窗必须有同色），150px 的图标空隙必让某个整窗落空；真按钮是实心长条不受影响。
2. **白卡列误报源是好友页的“白底样式”**：这位好友的资料页人物背景是整片纯白 + 浅色队徽水印，并非 v7 假设的“天空蓝场景”——右侧 74%~86% 探测带从上到下全是纯白，“纵向白色长列”天然成立，再怎么加长白色验证都区分不了两页。修复：换思路，给页面标记加**好友页锚点一票否决**——LOCAL TRADE（35%~43%）和 BATTLE（81%~86%）两枚图标窗都检测到青绿色（v7.1 的误报恰好证明了这排图标的颜色就落在 `GreenBtnSpec` 容差盒内）、且两窗之间 46%~55% 净空时，确认当前站在好友页，标记强制 False、直接放行进交易流程（放行安全：锚点成立 = 入口坐标点的就是 LOCAL TRADE 图标本尊）。详情页上按钮最宽到 ~42%，BATTLE 窗位置是白卡内容区、绝不会有青绿色，锚点恒为不成立，否决不可能吞掉真残留的详情页。

收尾关闭验证同样受益：点掉真 X 回到好友页后，即使探针在好友页再说谎，锚点否决也会把标记压成 False，“标记消失=关闭成功”的判定照常成立，不再依赖保险丝兜底。

**第九轮（v7.3）：开局判定反转 + 等待全面检测驱动（用户反馈）** —— 两个问题：① 使用方式本就要求“从好友页开始”、收尾又已经验证过“标记消失”，开局凭什么默认怀疑“详情页没关”？v5~v7.2 的三次开局误报全是这个怀疑式预检在好友页上说谎造成的。② `list settle 1/4 sec`、`return settle 1/5 sec` 这类纯等待太机械、逐秒刷屏。修复：

1. **轮前判定反转**：默认相信“此刻就该在好友页”，用好友页锚点做**正向确认**——锚点一出现立即放行（此时入口坐标点的就是刚确认过的 LOCAL TRADE 图标本尊，构造性安全），正常轮次开销接近零、一行日志。锚点超时（默认 10 秒）才转入旧的“标记三连确认”怀疑分支：真有残留详情页照旧安全退出；标记没被三连确认（如 BATTLE 变灰让锚点失效）则告警后放行，由 NEXT 超时兜底。开局检查从“证明没在详情页”变成“确认在好友页”，误报面从此只剩“锚点漏认好友页”一种，而它的后果只是多等 10 秒 + 走兜底，不再有任何点击风险。
2. **等待检测驱动化**：点 LOCAL TRADE 后的 4 秒盲等改为“锚点消失=已切页”检测（超时自动补点一次入口，比盲等更稳）；收尾后的 5 秒盲等删除，并入下一轮开头的“锚点就绪”检测（页面一回来立即开始）；删掉 `WaitSeconds` 和它的逐秒日志。典型每轮节省 6~8 秒。

**第十轮（v7.5）：好友页图标改版成彩色插画，锚点整窗漏认、开局误停** —— 实测日志：脚本一开始就在好友资料页，却连续打印 `等待好友页就绪（锚点未现）` 等满 10 秒，随后 `标记诊断[轮前]：绿长钮=无，白卡列=有(718,997)，好友锚点=不成立` 三连确认、安全退出，整轮没法跑。对照截图定位根因：**这排功能图标已从 v7.2 标定时的扁平青色改版成彩色插画**——LOCAL TRADE 是蓝色手机、REMOTE TRADE 是皮卡丘、BATTLE 是绿色场景。`GreenBtnSpec`（G≥168 且 B≤200 的黄绿盒）对蓝色手机图标整窗漏检，锚点第一条腿就断；锚点一断，白底/深色样式好友页的白色面板又让白卡列天然成立，于是按 v7.1 规则误停。

PR#10（v7.4，未合并）曾尝试“`FriendIconSpec = CloseSpec` 整体换色谱 + 图标行高度窗放宽到 73.5%~80.5%”，复盘发现三个问题，被本轮取代：

1. **单色盒换猜，顾此失彼**：`CloseSpec` 的 B 下限是 105，BATTLE 绿色场景插画的饱和绿（B 常 <105）会掉出盒外——修了蓝图标可能反丢绿图标，和被它替换的 `GreenBtnSpec` 漏蓝是同一类错误，而且没有任何实测色值支撑。
2. **高度窗放宽到 80.5% 圈进了图标文字标签**（~79.2% 起）：`CloseSpec` 容差盒含 132~155 的中性灰，正是文字抗锯齿的必经色；“REMOTE TRADE”标签比图标宽、左端伸进 46%~55% 间隙窗，一个像素就触发净空否决——锚点会在真好友页上被自家规则否决，等于复现要修的 bug。
3. **削弱“锚点在详情页恒不成立”的安全不变量且无论证**：`CloseSpec` 同时兼容青色界面文字和中性灰，详情页白卡上的文字可能点亮 BATTLE 窗；锚点既是页面标记的一票否决又是轮前放行依据，假阳后果是入口坐标盲点到 POWER UP / EVOLVE（本仓库定义的最高危事故）。

v7.5 的修法：① 图标窗找色改**三色谱并集**（`GreenBtnSpec` / `CloseSpec` / 新增 `FriendBlueSpec` 饱和蓝盒，任一命中即算图标在场），旧青色图标和新插画图标都覆盖；② 图标行高度窗 73.5%~78.8%，上沿略放宽、**下沿保持 78.8% 不动**把文字标签挡在窗外；③ 间隙窗右缘 55%→54%，离 REMOTE TRADE 图标左缘（实测 ~56.4%）让出安全边；④ 锚点新增**绿长钮硬安全闸**：检测带内验证出实心绿色长按钮（实心五连窗版，v7.2 已证明好友页上不可能存在）= 一定不是好友页——色谱再宽，“可点按钮的详情页”也在结构上不可能被认成好友页，同时 `DetailPageOpen` 的否决也永远吞不掉绿钮详情页；⑤ 锚点超时打印**锚点分解**日志（LT 窗/BT 窗/间隙 × 绿/青/蓝逐项命中 + 绿长钮闸状态），下次再漏认不用截图猜色、直接按窗校准；⑥ 轮前三连确认**分流**：含绿长钮才硬停（不可逆风险），只有白卡列则告警放行——v7.1 当年改成“确认即退出”防的就是绿钮误进化，绿钮探针本身就是该风险的直接证据，白卡-only 时入口坐标即使真点在灰按钮/满级详情页上也无副作用，会由 NEXT 超时安全兜底；锚点将来再漏认，代价从“整轮硬停 + 手动重跑”降为“多等 10 秒 + 两行告警”。

## 已知边界

- 如果**被交易的宝可梦本身是大面积黄绿色**（如绿毛虫系），模型恰好伸进 CONFIRM 检测带（左侧 38%~59% 高度）可能被误判成按钮；点了没变橙会自动重试并最终安全退出。把 `ConfY2` 调小即可。
- **好友页锚点依赖 LOCAL TRADE 和 BATTLE 两枚图标有可识别的颜色**：好友不在附近时 BATTLE 图标会变灰，锚点会不成立。后果是每轮开头把 `FriendReadyMaxSec`（默认 10 秒）等满再走兜底分支（v7.5 起：没确认出残留、或只确认出白卡列，都告警后放行；只有确认出绿长钮才安全退出），不会引入新的误点击。常驻这种场景可把 `FriendReadyMaxSec` 调小；本脚本的 LOCAL TRADE 流程本来就要求好友在附近。
- **锚点窗按“四按钮好友页”标定**（SEND GIFT / LOCAL TRADE / REMOTE TRADE / BATTLE 同行四枚）：没有 REMOTE TRADE 的三按钮布局图标位次不同，LOCAL/BATTLE 窗可能落空、间隙窗可能压在中间图标上，锚点会漏认。后果同上（多等 10 秒 + 告警放行），且入口坐标 (386,1555) 在三按钮布局下点的不一定是交易入口，会由 NEXT 超时安全退出。图标若再次改版换色同理——先看日志里的 `锚点分解` 行，哪个窗、哪个色谱没命中一目了然，对应微调 `FriendBlueSpec` 等色盒或 `Fri*` 区域即可。
- 特殊交易（幸运/异色/未登录图鉴）有额外弹窗，不在处理范围内；脚本会因等不到预期状态安全退出。
- 某一步识别不到时，优先用 IDE 取色器取实际颜色（注意显示的是 BGR），微调对应 `*Spec`；位置不对就调对应区域百分比。日志里每一步都有“等什么、等了多久”，能直接定位卡点。
